# CH+ 工具链的立场声明

## 我们反对什么

**收费的中文编程语言。**

易语言、火山中文编程、及一切把"中文编程"做成付费墙的产品。

它们做的事情很统一:把一门编程语言裹上"民族情怀"和"降低门槛"的外衣,然后按授权费、按坐席、按版本收费。程序员学了一门封闭语言,绑死在一个生态里,某天厂商涨价、跑路、停更,学的功夫和写的代码全打水漂。

这不是"中文编程",这是**用中文当锁定用户的工具**。中文不该是收费的借口,中文是汉语使用者的母语权利。

## 我们要什么

一门**开源、免费、跨平台、文本代码、git 友好**的中文编程语言工具链。

- **开源**:代码看得见,改得了,跑得通
- **免费**:个人学、公司用、教学用,不收一分钱
- **跨平台**:Mac / Linux / Windows 一份代码到处跑,不绑死任何系统
- **文本代码**:`.ch` 是 UTF-8 文本,git 能 diff、能 merge、能 review,不搞二进制封闭格式
- **工具链现代化**:cargo / uv 风格,`chplus new` 就能开干,不是上世纪的 IDE 套壳

## 关于当前许可证的诚实说明

本项目 fork 自 `abcdefgjh-li/chplus` v3.0.0,沿用了其 **AGPL-3.0 + Commons Clause 1.0** 协议。

我们理解原作者的用意:用 AGPL 防止商业闭源 fork,用 Commons Clause 堵死商用渠道,把"收费中文编程"这条路在许可证层就焊死。**这立场我们认同。**

但我们也诚实承认:AGPL 的强 copyleft 对非商业用户也不友好。你用 CH+ 写个公司内部工具,严格按 AGPL 解释也得开源,这确实过分。

这是当前的妥协,不是终态。我们的长期方向是:

1. **CLI wrapper 部分(`cli/`)** — 用 Rust 写,独立作品,不派生 C++ 解释器,法律上可单独挂宽松协议(MIT/Apache-2.0)。
2. **底层解释器部分(`chplus-lang/`)** — 暂时沿用原协议,不动 C++ 源码一个字。
3. **远期目标** — 用 Rust 做 clean-room 重写解释器,彻底摆脱 AGPL 束缚,改用对个人和商业都更友好的协议(MIT 或 Apache-2.0)。

这是工作量,不是立场问题。会做,但不骗你说马上能做。

## 我们不做的事

- **不做图形界面 IDE** — 这是另一件事,留给 VSCode 插件等独立项目
- **不做收费功能** — 不会有"Pro 版""企业版""付费插件"
- **不绑死平台** — 不靠 Windows DLL 锁生态,不靠 macOS 私有 API 制造差异
- **不卖情怀** — 中文是工具属性,不是营销卖点

## 给从易语言/火山迁移过来的人

如果你被收费中文编程坑过,这里有一条出路:

1. `cargo install` 装好 `chplus`,免费的
2. `chplus new 我的项目` 开个项目
3. 写 `.ch` 文件,中文关键字,git 能管
4. `chplus run` 跑起来

语法和易语言不完全一样,但思路相通。中文关键字降低的是阅读门槛,不是思考门槛 — 编程的本质还得自己学。

## 一句话

**中文编程不该是门生意。**
