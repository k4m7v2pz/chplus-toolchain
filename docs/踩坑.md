# CH+ 踩坑记录

给后续 agent 和 contributor 留的实地经验,避免重复踩坑。基于实际编码与测试整理。

---

## 一、CH+ 语言层踩坑(底层解释器限制)

### 1. 函数返回整型被误判为字符串

**症状**:
```
定义(整型) 平方(定义(整型) n) {
    返回 n * n;        // 报错:返回类型不匹配,期望整型实际字符串
}
```

**原因**:底层解释器的返回类型校验有 bug,数值运算的结果被当作字符串处理。

**绕法**:
- 关键数值逻辑写在主函数里**内联**,不抽成独立函数
- 真要抽函数,接受只能返回字符串,然后再转整型:`定义(整型) x = 字符串函数();`(字符串到整型隐式转换可用)

### 2. ANSI 转义码不识别

**症状**:`控制台输出("\033[2J\033[H")` 期望清屏,实际输出字面量 `033[2J033[H`。

**影响**:
- 不能清屏 → 游戏画面只能靠换行刷,界面难看
- 不能着色 → 没法用红绿区分胜负
- 不能移动光标 → 不能就地更新棋盘

**绕法**:接受换行刷屏,或放弃 TUI 美化走文字交互。底层不修没法绕。

### 3. `系统命令()` 函数未定义

**症状**:调 `系统命令("echo hi")` 报"函数未定义"。

**影响**:不能调外部 shell,失去绕底层能力缺失的一条路(比如调 python 画图、调 curl 起服务)。

**绕法**:无。底层就没实现这个函数,等修。

### 4. 取余 `%` 在复杂表达式里行为异常

**症状**:`控制台输出(7 % 3)` 输出错误结果(不是 1)。

**绕法**:把取余结果先存到变量,再拼接输出:
```
定义(整型) r = 7 % 3;
控制台输出(r);        // 正确输出 1
控制台输出(7 % 3);    // 错误
```

### 5. 函数参数语法必须完整 `定义(类型) 名字`

**错误写法**:
```
定义(整型) 加(整型 a, 整型 b) { ... }   // 报错:函数参数必须使用 '定义(类型) 名字' 语法
```

**正确写法**:
```
定义(整型) 加(定义(整型) a, 定义(整型) b) { ... }
```

每个参数都要完整 `定义(类型) 名字` 形式,不能省略 `定义`。

### 6. `否则如果` 必须用中文,不能用 `else if`

**错误**:`else if (x > 0) { ... }`
**正确**:`否则如果 (x > 0) { ... }`

CH+ 关键字全中文,混用英文会报"语句必须以分号结束"之类奇怪错误。

### 7. 字符串自动转整型可用,反向需手动

```
定义(字符串) s = "42";
定义(整型) n = s;          // ✓ 隐式转整型
控制台输出(n + 1);          // 输出 43
```

但整型拼接到字符串时会出问题(见下条),需注意。

### 8. 字符串拼接 + 整型输出有歧义

`控制台输出("数字" + 5)` 行为不稳,推荐:
```
定义(整型) x = 5;
控制台输出("数字 " + x);    // 用变量拼接,稳
```

不要在拼接表达式里做算术:`控制台输出("结果" + (a + b))` 会出怪事。

### 9. 全局可变变量跨函数访问不可靠

**症状**:函数外定义的 `seed` 在函数内修改后,主函数读不到原值。

**绕法**:把状态当参数显式传递:
```
定义(整型) 下一个随机(定义(整型) s) {
    返回 (s * 11035 + 12345) % 32768;    // 但这又会撞坑 1
}
定义(空类型) 主函数() {
    定义(整型) s = 42;
    s = (s * 11035 + 12345) % 32768;     // 内联到主函数,稳
}
```

### 10. 没有 `返回` 多值/元组,不能解构

CH+ 不支持多返回值。要返回多个值只能:
- 拼成字符串返回,接收方再解析(笨但能跑)
- 用结构体打包(结构体可用,见下条)

## 二、CH+ 能做什么(实测可用清单)

| 能力 | 写法 | 备注 |
|---|---|---|
| 整型/小数/字符串/布尔变量 | `定义(整型) x = 1;` | ✓ |
| 输入输出 | `控制台输出() / 控制台输入()` | ✓ 输入是阻塞的 |
| 算术 | `+ - * / % ^` | ✓ `%` 见坑 4 |
| 比较 | `== != < > <= >=` | ✓ |
| 逻辑 | `&& \|\| !` | ✓ |
| 条件 | `如果 / 否则如果 / 否则` | ✓ 必须中文 |
| 循环 | `对于 / 当` | ✓ C 风格 |
| 函数 | `定义(返回类型) 名(定义(类型) a)` | ✓ 见坑 1、5 |
| 数组 | `定义(整型) arr[10];` | ✓ |
| 结构体 | `定义(结构体) 名 { 整型 x; }` | ✓ |
| 字符串方法 | `s.length()` | ✓ |
| 文件 I/O | `导入("ch_Lib/file.ch");` | ✓ 含读写、追加、重定向 |
| 数学库 | `导入("ch_Lib/math.ch");` | ✓ PI、三角、对数、幂 |
| 字符串库 | `导入("ch_Lib/string.ch");` | ✓ |
| 导入其他 .ch | `导入("path/x.ch");` | ✓ 路径相对 CWD(坑 11) |

## 三、CH+ 工具链踩坑(cli 层)

### 11. `导入()` 路径相对 CWD,不相对源文件

**症状**:库 A 写 `导入("ch_Lib/string.ch")`,在 A 自己项目根目录跑没问题;A 被作为依赖拉到项目 B 的 `chplus_modules/A/` 后,B 项目跑时这个导入还按 B 的 CWD 找 `ch_Lib/string.ch`,找不到就报错。

**绕法**:目前约定 — 库只导入随工具链分发的 `ch_Lib/`(B 项目根目录下)和自己同目录文件,不要导入 `chplus_modules/` 里别的依赖。依赖图交给 `chplus.toml` + `chplus add` 管。

### 12. TOML 不支持裸中文 key

**症状**:`某库 = { git = "...", tag = "..." }` 解析失败,报 "invalid key"。

**原因**:TOML RFC 规定 bare key 只能 `A-Za-z0-9_-`,中文必须加引号。

**绕法**(已在 `add.rs` 实现):写 toml 时检测中文自动加引号 → `"某库" = { ... }`。

### 13. `--path` 的值会被误认为位置参数

**症状**:`add 别名 --path /abs/path` 被误判为同时指定了 git url 和 --path(因为 `/abs/path` 不以 `-` 开头被 `positional` 收集)。

**绕法**(已修):互斥检查时排除 `--path` 自身的值。

### 14. fmt 会在 `(` 前加空格、删尾换行

**应对**:`chplus new` 模板已预格式化迎合它(模板里就是 `定义 (空类型) 主函数() {` 这种带空格形态),所以新项目 `fmt --check` 直接过。手写代码被 fmt 改过后形态会变,不要惊讶。

### 15. fmt 默认会覆盖源文件

**应对**:CLI 所有调用底层都加 `-n` 禁掉。只有 `chplus fmt`(显式格式化)才允许覆盖。

### 16. `-p` flag 已废弃

不要用,行为不定。

### 17. fmt/check 时 stdout 有噪音

底层会打 "文件已自动格式化" 之类提示。CLI 调用丢 stdout/stderr,只看文件差异或 exit code。

## 四、架构与策略踩坑

### 18. wrapper 模式不要破坏

CLI 是 Rust wrapper,通过子进程调底层 `chplus_core`。**不要改 C++ 解释器源码** — 这会触发 AGPL+Commons Clause 的衍生作品条款扩散,且与你"只做工具链不做语言"的定位冲突。

底层有 bug?绕过去,不修。除非哪天做 clean-room 重写。

### 19. 不要引重量级依赖

- 拉依赖用 `Command::new("git")` 而非 `git2` crate(后者编译慢、二进制大)
- watch 用 poll 不用 notify 的事件去抖(实际 poll 在小项目上够用,代码更少)
- CLI 全依赖就 anyhow + serde + toml + notify,控制在最小

### 20. AGPL+Commons Clause 是当前妥协,不是终态

诚实承认对非商用也不友好。远期 clean-room 重写底层后换 MIT/Apache-2.0。不要现在就违反,也不要假装它是完美协议。

## 五、测试踩坑

### 21. 自动化测试 `echo 输入` 喂 stdin 时当心死循环

`当 (guess != answer)` 这种循环,如果 `echo 0` 喂进去但 answer 不是 0,会一直 `控制台输入` 拿不到东西卡死或死循环。

**绕法**:测试时用 `timeout 5 chplus run xxx.ch`,超时即判异常;或喂正确答案直接验证成功路径。

### 22. `cargo check` 编译警告里的 dead code 不要急着删

manifest/lock 的方法像 `git_ref()`、`is_path()`、`local_name()` 现在不一定被调用,但后续命令会用到。删了又要重加。

## 六、给后续 agent 的工作建议

1. **改 CLI 前**先 `cd cli && cargo check`,改完再 `cargo build`,跑一遍冒烟测试
2. **改游戏前**先 `chplus lint` 看符号结构,再用 `echo 答案 | chplus run 主函数.ch` 测交互
3. **底层有 bug 时**优先找绕法,不要动 `chplus-lang/`
4. **新命令的参数解析**用 `take_value` + `positional` 分离 flag 和位置参,参考 `add.rs`
5. **写游戏**别抽工具函数(坑 1),逻辑内联到主函数;UI 别想 ANSI(坑 2),走换行刷
6. **写文档**坦白说现状,不夸大能力,新手读到"待实现"比读到"很强大"跑起来发现不能用强
